# MVP Philosophy

**The core loop first, everything else never.** Every MVP has one job: deliver on its core promise. Identify the shortest path from input to value and make that path work reliably before touching anything else. An app that handles every edge case but fails at its main job is not a product.

**Only build what you'd demo.** If you wouldn't show a feature to someone because it's invisible or only triggers on failure, it's not MVP scope. The demo is the happy path. Build that until it's embarrassingly good, then expand.

**Tunnel vision is a feature, not a flaw.** It is natural — especially as a developer — to immediately enumerate what could go wrong and feel compelled to solve it. Resist this. Edge cases, error handling, and defensive logic are noise until the happy path is proven. Solving the wrong problems first is the most common way MVPs never ship.

**Your first user is you.** For the earliest version of anything, build for a user whose behavior you know exactly: yours. You'll know immediately when something is wrong. Solve for that before solving for unknown users.

**The car has to start.** Safety features matter. Error handling matters. Robustness matters. But not before the thing turns on.
